## 오후 세션 — 프로파일링 & 코드 최적화 타당성 검토

> 오전(MPI 병렬화)과 별개. 팀 조언 "Profile → fixture → 최적화" 순서를 그대로 따름.

### 한 일 (오후)
- **최적화 전에 프로파일링부터**: 솔브 계산에 영향 없는(비침습) **4계층 계측**을 붙임 — 단계별(Allrun 타임스탬프), 솔버 내부(로그 파싱), 파이썬 구간(perf_counter). 계측 유무로 온도 결과가 소수 5자리까지 동일함을 실측(=fixture로 재사용 가능).
- **4코어 baseline(1,693초) 분해 결과**: 솔브가 벽시계의 **98%**, 그중 **fluid(냉각수) 영역이 ~98%+**. 최비용 압력식 `p_rgh`는 **이미 GAMG(멀티그리드)=최적 솔버**.
- **교훈**: 선형반복 *횟수*는 벽시계의 나쁜 대리지표. 고체 온도식 `h`(tube+mb)가 전체 반복의 72%지만 실제 시간은 ~1%.
- **답-보존 config 최적화 2건을 실제 4코어로 실행**(각 ~28분):
  - ① 고체 `h` 솔버 PCG→**GAMG**: 안정(Δ<0.0002 °C) but **−1.4%**(노이즈 수준)
  - ② 분해 `simple(4×1×1)`→**hierarchical(2 2 1)**: 안정(Δ<0.002 °C) but **−2.0%**(노이즈 근처)
  - run 간 노이즈 바닥이 ~1%(변경 없이도 1679/1693/1692초)라 **둘 다 유효 단축 아님 → 폐기·원복**.
- 반복수 축소(가장 큰 지렛대)는 미러가 2000 iter에서 캘리브레이션됐고 fluid max가 아직 수렴 중이라 **정확도/안정성 리스크로 제외**.
- **소스 접근 타당성 조사**: 이미지에 OpenFOAM **소스 트리 + wmake + gcc** 있음 → 재컴파일 기술적으로 가능. 그러나 솔버 앱은 **773줄 오케스트레이션**뿐이고 실제 계산은 **18MB+ 성숙 라이브러리**(`libfiniteVolume` 등) 안. **HEAT의 "이중 포문"에 해당하는 개선 여지가 앱 레벨에 없음** + **GPLv3 배포 의무** + fork 유지부담.

### 판단
- **고정 4코어에서 결과를 보존하며 벽시계를 유효하게 줄이는 config 레버는 없다.** 코드 최적화는 HEAT만큼의 ROI를 기대하기 어렵다.
- 팀 의견("가속 여지가 없으면 Ansys가 나을 수도 있으니 이 판단을 먼저")에 정합. **코드 최적화 < 서로게이트** 쪽으로 기우는 중 → 1~2일 더 시도 후 결정.
- 팀 질문 반영: 온도 캘리브레이션(y+·열유속 매핑)은 **속도가 아니라 정확도** 이슈이고, 샘플이 바뀌면 다시 모델링해야 하는 **샘플 종속** 개선이라 일반화가 약함 → **지금 우선순위에서 내림**.
